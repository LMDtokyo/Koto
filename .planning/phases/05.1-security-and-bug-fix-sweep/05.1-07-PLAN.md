<objective>
Close SC-8 (SC-8 gates: "Killing + restarting the Android app mid-session does not break subsequent message decryption").

Root cause: `CryptoManager.init()` restores identity + signedPrekey + kyberPrekey from `AccountPrefs`, but not one-time prekeys. The Rust side's `InMemSignalProtocolStore` is an in-memory HashMap<PreKeyId, PreKeyRecord>. On process kill, the map dies. A PreKeySignalMessage arriving after restart references an OTK id whose record is gone → `InvalidPreKeyId` → decryption fails forever.

Per RESEARCH Pattern 3: libsignal's `message_decrypt_prekey` automatically removes the OTK from its in-memory store when a PreKey message is processed. So the durability contract is: (a) persist every generated OTK to Room at generation time, (b) on app startup, reload all `used=false` rows into the in-memory store via `cryptoManager.loadPrekeys(...)`, (c) rely on libsignal's own cleanup to prune the in-memory map — the Room row stays with `used=false` indefinitely (cheap: ~64 bytes × 100 keys = 6.4 KB).

Output:
  - New Room entity + DAO
  - Room schema version v3 → v4 with MIGRATION_3_4
  - Hilt module exposes `OneTimePreKeyDao`
  - `CryptoManager.generateAndSaveKeys` persists generated OTKs
  - `CryptoManager.init` loads unused OTKs before returning
  - Fill in the Wave 0 instrumented test to assert round-trip
  - SC-13: `./gradlew :app:assembleDebug` passes with no new lint errors

**uniffi binding fact (confirmed against `crypto/src/lib.rs:66`):** The `RegistrationBundle` Rust struct declares `pub prekeys: Vec<PreKeyData>` (NOT `one_time_prekeys`). uniffi's default camelCase mapping leaves single-word snake identifiers unchanged, so the Kotlin field is `bundle.prekeys`. Do not use `bundle.oneTimePrekeys` — that field does not exist. (Source: crypto/src/lib.rs lines 55-69.)
